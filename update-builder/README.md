# Kronos Update Builder Guide

## Overview

The Korg Kronos update mechanism is driven by the [UpdateOS](../docs/modules/UpdateOS.md).
When the user initiates an OS update from the Kronos front panel, `UpdateOS` mounts a USB stick, reads `install.info`, validates the package, and runs the embedded shell scripts.

This document records the studied signature algorithm and explains how to use
`update_builder.py` to produce your own, valid, update packages.

When UpdateOS runs, it looks for a USB stick containing a file named `install.info` in the root folder.
If found, UpdateOS mounts the USB at `/mnt/updaterSource/` and reed all files from there.

## The install.info Format
This is a plain `KEY=VALUE` text file, and UpdateOS strips leading/trailing whitespace and ignores
lines beginning with `#` when processing it. Example:
```
VERSION=3.2.1
SOURCE=KronosOS.tar.gz
PRETARSCRIPT=pretar.sh
POSTTARSCRIPT=posttar.sh
SIGNATURE=<40-char SHA-1 hex>
```

Rules enforced by UpdateOS:

- `VERSION` — required to be **present**; any value is valid and accepted. UpdateOS never
  compares the value against the installed OS version — it is stored for display
  purposes only (shown on the front panel during the update). 
  If the field is absent, UpdateOS writes `"Failed. Bad install.info - VERSION missing"` to
  `/tmp/updater_status` and exits with code 9.
- `SOURCE` — required; the tar.gz that is extracted to the filesystem.
- `PRETARSCRIPT` — shell script run **before** tar extraction.
- `POSTTARSCRIPT` — shell script run **after** tar extraction.
- `SIGNATURE` — required **if and only if** at least one script field is present.
  If no scripts are present, the field is ignored / not checked.

## Signature Algorithm

UpdateOS has a function `VerifyScriptsSignature` which validates the SIGNATURE line  by comparing it to:

```
SIGNATURE = lowercase_hex( SHA1( pretar_bytes + posttar_bytes + UPDATER_KEY ) )
```
where `UPDATER_KEY = 13 d0 af ef  e0 3c 9b 92  16 2f ae ff  77 53 55 e1`

- Scripts are fed into the SHA-1 digest **in order**: pretar first, posttar second.
- If only one script is present, only that script's bytes are fed (the other is skipped).
- The 16-byte `UPDATER_KEY` is always appended last, after all script bytes.
- The result is lowercase hex-encoded (40 characters).


## update_builder.py — Usage Guide

### Prerequisites

Python 3.8+ (uses only stdlib: `hashlib`, `argparse`, `pathlib`).

### To create `install.info` from scratch

```bash
./update_builder.py /path/to/update_folder \
    --version 3.2.1 \
    --source MyUpdate.tar.gz \
    --pretar pretar.sh \
    --posttar posttar.sh
```

All four `--version`, `--source`, `--pretar`, `--posttar` flags override whatever is
currently in `install.info` (or supply values when no file exists yet).

To omit a script entirely, pass an empty string:

```bash
./update_builder.py /path/to/update_folder --pretar '' --posttar ''
```

This produces an `install.info` with no scripts and no SIGNATURE field (UpdateOS will
accept this).



### To Re-sign an existing `install.info`

If you already have an `install.info` with VERSION, SOURCE, PRETARSCRIPT and/or
POSTTARSCRIPT set, just point the script at the folder:

```bash
./update_builder.py /path/to/update_folder
```

The script reads the existing fields, computes the correct SIGNATURE, and rewrites
`install.info` in-place.



### 3. To Preview without writing

Append `--dry-run` to any invocation to print computed values without touching any file:

```bash
./update_builder.py /path/to/update_folder --dry-run
```

### Example output

```
VERSION      : 3.2.1
SOURCE       : MyUpdate.tar.gz
PRETARSCRIPT : pretar.sh
POSTTARSCRIPT: posttar.sh
SIGNATURE    : 8844b641ee9ba99a35897dc961885c82850d08c0
```

### Generating a minimal no-script update

If your update is just a tar.gz with no pre/post scripts (e.g. a pure data drop):

```bash
./update_builder.py /tmp/my_update \
    --version 3.2.1 \
    --source MyPayload.tar.gz \
    --pretar '' \
    --posttar ''
```

Produces:
```
VERSION=3.2.1
SOURCE=MyPayload.tar.gz
```

No SIGNATURE line — UpdateOS will skip signature verification entirely.

## Building a Complete Update Package

A valid update USB stick needs:

```
/                        ← root of USB (FAT or ext2)
├── install.info         ← generated by update_builder.py
├── <SOURCE>.tar.gz      ← the payload (extracted to Kronos filesystem)
├── pretar.sh            ← (optional) shell script, runs before extraction
└── posttar.sh           ← (optional) shell script, runs after extraction
```

Steps:

1. Prepare the payload tar.gz (contents relative to `/` on the Kronos filesystem).
2. Write `pretar.sh` and/or `posttar.sh` if needed.
3. Run `update_builder.py` to generate `install.info`.
4. Copy all files to a FAT-formatted USB stick.
5. Insert into Kronos and trigger update from the front-panel menu.

UpdateOS mounts the USB at `/mnt/updaterSource/`, so `install.info` must be at the root
of the stick.

## Notes on the `kronos_rooting` Update Technique

The rooting update (by `linux-kronos` on GitHub) uses exactly this mechanism. Its
`pretar.sh` installs dropbear and other utilities into locations that survive the
integrity checks run by `loadoa` and `loadmod.ko`. The `install.info` signature is
computed with the same key and algorithm documented here.
